Manual Esencial de Git y Control de Versiones

Una guía completa y moderna para dominar Git, el sistema de control de versiones distribuido que impulsa el desarrollo de software actual.

Fecha: 4 de julio de 2025 | Autor: Kevin Aráuz

1. ¿Qué es Git y por qué usarlo?

Git es un software de control de versiones distribuido, diseñado por Linus Torvalds. Imagínalo como una "máquina del tiempo" para tu código. Te permite guardar diferentes versiones de tu proyecto, volver a estados anteriores, trabajar en equipo sin sobreescribir el trabajo de otros y experimentar con nuevas ideas de forma segura.

"Git es un software de control de versiones diseñado por Linus Torvalds, pensando en la eficiencia, la confiabilidad y compatibilidad del mantenimiento de versiones de aplicaciones cuando estas tienen un gran número de archivos de código fuente."

Definición adaptada de Wikipedia

2. Conceptos Fundamentales de Git

Antes de usar los comandos, es vital entender el vocabulario. Estos son los pilares sobre los que se construye todo en Git.

Repositorio (Repository)

Es la carpeta de tu proyecto "con superpoderes". Contiene todos tus archivos y, lo más importante, una subcarpeta oculta llamada .git. Esta carpeta es la base de datos que almacena todo el historial de cambios de tu proyecto.

Commit

Es una "instantánea" o un punto de guardado en la historia de tu repositorio. Cada commit guarda el estado exacto de todos tus archivos en un momento dado. Es una unidad atómica de cambio que incluye un mensaje que describe qué se hizo.

Rama (Branch)

Imagina una línea de tiempo de tu proyecto. Una rama es simplemente otra línea de tiempo que se deriva de la principal. Te permite trabajar en una nueva función o corregir un error de forma aislada, sin afectar la versión estable del proyecto.

Fusionar (Merge)

Es el acto de tomar los cambios de una rama (por ejemplo, la de tu nueva función) e integrarlos de nuevo en otra rama (generalmente la principal, llamada main). Git es increíblemente bueno resolviendo estas fusiones automáticamente.

HEAD

Es simplemente un puntero que indica en qué punto de la historia de tu proyecto te encuentras. Normalmente, HEAD apunta a la última confirmación de la rama en la que estás trabajando actualmente.

Remoto (Remote)

Es una versión de tu repositorio que está alojada en un servidor en la nube (como GitHub o GitLab). Sirve como una copia de seguridad centralizada y es la forma principal de colaborar con otros desarrolladores.

3. Instalación de Git

Linux

Usa el gestor de paquetes de tu distro. En Debian/Ubuntu: sudo apt install git. En Fedora: sudo dnf install git.

Windows & macOS

Descarga el instalador oficial desde git-scm.com. El asistente te guiará en el proceso.

Verificación

Tras instalar, abre tu terminal y ejecuta el siguiente comando para confirmar que todo está correcto:

git --version

4. Configuración Inicial

Antes de empezar, configura tu identidad. Esta información se usará en cada commit que realices.

Establece tu nombre de usuario y correo electrónico. Estos comandos solo necesitan ejecutarse una vez por computadora.

git config --global user.name "Tu Nombre"
git config --global user.email "tu.email@ejemplo.com"

La opción --global aplica esta configuración a todos tus proyectos.

  • Editor de Texto: git config --global core.editor "code --wait" (Para VS Code).
  • Rama Principal por Defecto: git config --global init.defaultBranch main.
  • Alias: Crea atajos para comandos largos. Ej: git config --global alias.co checkout.
  • Verificar Configuración: git config --list.

5. Creando Repositorios

Inicializar un Repositorio (git init)

Convierte un directorio existente en un repositorio de Git.

cd mi-proyecto
git init

Esto crea una carpeta oculta .git que contiene todo el historial y la configuración.

Clonar un Repositorio (git clone)

Obtén una copia local de un proyecto existente desde un servidor remoto (como GitHub).

git clone <URL_DEL_REPOSITORIO>

Esto descarga todo el proyecto y su historial completo.

6. El Flujo de Trabajo Básico de Git

Para guardar cambios en Git, los archivos viajan a través de varias etapas. Entender este flujo es crucial para usar Git de manera efectiva.

1. Directorio de Trabajo (Working Directory)

Esta es tu carpeta de proyecto en tu computadora. Contiene todos los archivos que puedes ver y editar directamente. Cuando haces un cambio, como escribir código o modificar un texto, lo haces aquí. En esta etapa, Git sabe que los archivos existen, pero no está rastreando activamente los cambios que no has preparado.

2. Área de Preparación (Staging Area)

Piensa en esto como una sala de espera o un borrador para tu próximo punto de guardado. Antes de hacer un commit, debes "preparar" los cambios que quieres incluir usando el comando git add. Esto te da un control preciso sobre qué cambios se guardarán, permitiéndote agrupar modificaciones relacionadas en un solo commit, incluso si has trabajado en varios archivos.

3. Repositorio Local (.git)

Esta es la base de datos de Git en tu propia máquina, ubicada en la carpeta oculta .git. Cuando ejecutas git commit, todos los cambios que estaban en el Staging Area se guardan permanentemente aquí como una nueva "instantánea" en el historial de tu proyecto. Todo este historial vive en tu computadora, por lo que puedes trabajar sin conexión.

4. Repositorio Remoto (Remote)

Es una versión de tu repositorio alojada en un servidor externo (como GitHub). Sirve como un punto central para sincronizar tu trabajo y colaborar con otros. Usas git push para enviar tus commits locales al remoto y git pull para traer los cambios de otros a tu repositorio local.

El Viaje de tus Cambios

El siguiente diagrama ilustra cómo tus cambios viajan a través de estas etapas usando los comandos principales de Git.

7. El Archivo .gitignore: Ignorando Archivos

No todos los archivos de tu proyecto deben ser guardados en el repositorio. El archivo .gitignore le dice a Git qué archivos o carpetas debe ignorar por completo.

¿Por qué ignorar archivos?
  • Dependencias: Carpetas como node_modules pueden contener miles de archivos que se pueden reinstalar fácilmente.
  • Archivos compilados: Código que se genera a partir de tu código fuente (ej. archivos .class en Java).
  • Archivos de entorno: Archivos como .env que contienen contraseñas o claves de API secretas. ¡Nunca deben subirse a un repositorio público!
  • Archivos del sistema operativo: Archivos como .DS_Store en macOS o Thumbs.db en Windows.
Ejemplo de un .gitignore

Simplemente crea un archivo llamado .gitignore en la raíz de tu proyecto y añade los patrones de los archivos o carpetas a ignorar.

# Ignorar archivos de log
*.log

# Ignorar carpeta de dependencias de Node.js
node_modules/

# Ignorar archivos de entorno
.env

# Ignorar carpeta de build
/dist
/build

# Ignorar archivos del sistema operativo
.DS_Store
Thumbs.db

8. Ramificación y Fusión

Las ramas son la característica estrella de Git. Permiten trabajar en nuevas funcionalidades o arreglos de forma aislada sin afectar la línea principal de desarrollo (main).

  • git branch: Lista todas las ramas. La activa se marca con un *.
  • git checkout -b nueva-rama: Crea una nueva rama y se cambia a ella.
  • git checkout main: Vuelve a la rama principal.
  • git merge nueva-rama: (Estando en main) Fusiona los cambios de nueva-rama en main.
  • git branch -d nueva-rama: Elimina la rama una vez fusionada.

9. Colaboración con Remotos

Los repositorios remotos (alojados en servidores como GitHub, GitLab o Bitbucket) son la clave para colaborar con otros y mantener una copia de seguridad de tu trabajo.

  • git remote -v: Muestra los remotos configurados. Por defecto, origin es el servidor desde el que clonaste.
  • git fetch origin: Descarga toda la información del remoto (nuevos commits, nuevas ramas) pero NO la fusiona con tu trabajo local. Te permite ver qué hay de nuevo antes de integrar.
  • git pull origin main: Es un atajo para git fetch seguido de git merge. Trae los últimos cambios de la rama main del remoto y los fusiona con tu rama local actual.
  • git push origin main: Sube tus commits locales de la rama main al repositorio remoto para compartirlos.

Pull Requests (o Merge Requests) son el corazón de la colaboración moderna. En lugar de empujar cambios directamente a la rama main, creas una "solicitud de extracción" en plataformas como GitHub. Esto permite que otros revisen tu código, dejen comentarios y lo aprueben antes de que se fusione, garantizando la calidad del código.

10. Conceptos Avanzados y Flujos de Trabajo

A medida que tus proyectos crecen, necesitarás estrategias más sofisticadas. Aquí exploramos algunos conceptos y flujos de trabajo clave.

Git Flow es un modelo de ramificación estricto diseñado para proyectos con ciclos de lanzamiento programados. Utiliza varias ramas con roles específicos:

  • main: Contiene el código de producción. Solo se fusiona desde ramas de release o hotfix.
  • develop: Es la rama de integración principal para las nuevas características. Aquí es donde se concentra el desarrollo diario.
  • feature/*: Se crean a partir de develop para trabajar en nuevas funcionalidades. Una vez terminadas, se fusionan de nuevo en develop. (Ej: feature/login-usuario).
  • release/*: Se crean a partir de develop cuando se acerca una nueva versión. Permiten preparar el lanzamiento (corrección de errores menores, documentación) sin detener el desarrollo de nuevas características en develop. Una vez listas, se fusionan tanto en main como en develop.
  • hotfix/*: Se crean a partir de main para corregir errores críticos en producción. Se fusionan tanto en main como en develop para asegurar que la corrección esté en futuras versiones.

Tanto merge como rebase sirven para integrar cambios de una rama a otra, pero lo hacen de formas muy diferentes.

git merge:

Es como decir: "Junta la historia de estas dos ramas". Crea un nuevo "commit de fusión" que une los historiales. Esto preserva la historia exactamente como ocurrió, pero puede resultar en un gráfico de commits desordenado y difícil de leer si muchas personas trabajan en paralelo.

git rebase:

Es como decir: "Toma mis cambios y ponlos al final de la otra rama". En lugar de crear un commit de fusión, rebase re-escribe el historial del proyecto moviendo tus commits para que aparezcan después del último commit de la rama de destino. El resultado es un historial lineal y limpio.

Regla de oro: Nunca uses git rebase en ramas públicas (como main o develop) que otros puedan estar usando. Re-escribir el historial compartido puede causar serios problemas. Es seguro usarlo en tus propias ramas de funcionalidades locales antes de fusionarlas.

Un conflicto ocurre cuando intentas fusionar dos ramas que han modificado la misma parte del mismo archivo. Git no sabe cuál versión conservar, así que te pide que decidas.

Cuando esto sucede, Git marcará el archivo en conflicto con marcadores especiales:

<<<<<<< HEAD
// Código de tu rama actual (ej. main)
=======
// Código de la rama que estás fusionando
>>>>>>> nombre-de-la-otra-rama

Para resolverlo:

  1. Abre el archivo en tu editor de código.
  2. Busca los marcadores <<<<<<<, =======, y >>>>>>>.
  3. Edita el archivo para dejar la versión final que deseas, eliminando los marcadores de Git.
  4. Guarda el archivo.
  5. Usa git add <archivo-resuelto> para marcar el conflicto como resuelto.
  6. Continúa con el proceso con git commit (si fue un merge) o git rebase --continue (si fue un rebase).

11. Comandos Básicos de la Terminal (Git Bash)

Aunque Git tiene sus propios comandos, te moverás por tu sistema de archivos usando una terminal. Git Bash en Windows emula una terminal de Linux. Aquí tienes los comandos más comunes para navegar.

  • pwd (Print Working Directory): Muestra la ruta completa del directorio (carpeta) en el que te encuentras actualmente.
  • ls (List): Enumera todos los archivos y carpetas visibles en tu directorio actual.
  • ls -a (List All): Similar a ls, pero también muestra los archivos y carpetas ocultos (aquellos que empiezan con un punto, como .git).
  • cd <nombre-carpeta> (Change Directory): Te permite moverte a una carpeta específica. Ejemplo: cd mi-proyecto.
  • cd ..: Te mueve un nivel hacia arriba en la jerarquía de carpetas (a la carpeta contenedora).
  • mkdir <nombre-carpeta> (Make Directory): Crea una nueva carpeta vacía con el nombre que especifiques.
  • clear: Limpia la pantalla de la terminal, ocultando todos los comandos y salidas anteriores para tener una vista limpia.

12. ¡Manos a la Obra! Prácticas Guiadas

La mejor forma de aprender es haciendo. Sigue estas prácticas en tu terminal para afianzar los conceptos clave de Git.

Vamos a empezar desde cero: crearemos una carpeta para nuestro proyecto y la convertiremos en un repositorio de Git.

  1. Crea una carpeta para el proyecto:
    mkdir mi-primer-repo
  2. Navega dentro de la nueva carpeta:
    cd mi-primer-repo
  3. Inicializa Git aquí:
    git init

    Verás un mensaje como: Initialized empty Git repository in /ruta/a/tu/proyecto/mi-primer-repo/.git/. ¡Felicidades, ya tienes un repositorio!

  4. Verifica que se creó la carpeta .git:
    ls -a

    Este comando te mostrará la carpeta oculta .git, que es el cerebro de tu repositorio.

Ahora que tenemos un repositorio, vamos a crear un archivo y a guardar nuestra primera "fotografía" del proyecto.

  1. Crea un archivo: El comando touch crea un archivo vacío.
    touch index.html
  2. Revisa el estado del repositorio:
    git status

    Git te informará que tienes "Untracked files" (archivos no rastreados). Esto significa que Git ve el archivo, pero no está vigilando sus cambios todavía.

  3. Añade el archivo al Staging Area: Le decimos a Git que queremos incluir este archivo en nuestro próximo guardado.
    git add index.html
  4. Revisa el estado de nuevo:
    git status

    Ahora el archivo aparece bajo "Changes to be committed" (cambios listos para ser confirmados). ¡Está en el área de preparación!

  5. Guarda los cambios en el repositorio:
    git commit -m "feat: Crea archivo index.html inicial"

    ¡Has hecho tu primer commit! El mensaje (-m) es crucial para describir qué hiciste.

En un proyecto real, trabajarás con muchos archivos. Veamos cómo manejarlos.

  1. Crea dos archivos más:
    touch about.html contact.html
  2. Revisa el estado:
    git status

    Verás que ambos archivos aparecen como "Untracked".

  3. Añade todos los archivos nuevos o modificados al Staging Area de una vez:
    git add .

    El punto . es un atajo para "todo en el directorio actual".

  4. Confirma los cambios:
    git commit -m "feat: Agrega las páginas de about y contact"

El Staging Area te permite crear commits atómicos y limpios, incluyendo solo los cambios que deseas. Vamos a verlo en acción.

  1. Modifica dos archivos:
    echo "Página principal" > index.html
echo "Acerca de nosotros" > about.html
  2. Revisa el estado:
    git status

    Verás que Git lista ambos archivos como "modified".

  3. Prepara solo el cambio de `index.html`:
    git add index.html
  4. Revisa el estado de nuevo: ¡Esta es la parte importante!
    git status

    Observa cómo ahora tienes dos secciones: "Changes to be committed" (con index.html) y "Changes not staged for commit" (con about.html).

  5. Crea un commit solo con lo preparado:
    git commit -m "docs: Actualiza contenido de la página de inicio"
  6. Comprueba el estado final:
    git status

    Git te mostrará que el cambio en about.html sigue pendiente de ser confirmado. ¡Has creado un commit limpio solo con los cambios que querías!

Vamos a simular la creación de una nueva funcionalidad en una rama separada para no afectar nuestro trabajo principal.

  1. Crea y cámbiate a una nueva rama:
    git checkout -b agregar-estilos
  2. Crea un archivo de CSS en la nueva rama:
    touch styles.css
  3. Añade y confirma el nuevo archivo en esta rama:
    git add styles.css
git commit -m "feat: Agrega hoja de estilos"
  4. Regresa a la rama principal:
    git checkout main
  5. Lista los archivos:
    ls

    ¡Magia! El archivo styles.css ha desaparecido, porque solo existe en la rama agregar-estilos.

  6. Fusiona la nueva funcionalidad: Trae los cambios de la otra rama a main.
    git merge agregar-estilos
  7. Lista los archivos de nuevo:
    ls

    Ahora styles.css aparece en la rama main. Has integrado exitosamente la nueva funcionalidad.

  8. Limpia la rama ya fusionada:
    git branch -d agregar-estilos
